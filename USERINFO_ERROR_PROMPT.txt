CONTEXT
=======
Repository: eSignet (MOSIP OIDC identity platform, Spring Boot 3.2.3, Java 21).
Endpoint under review: GET /v1/esignet/oidc/userinfo
Spec reference: OpenID Connect Core 1.0 — UserInfo Error Response
  https://openid.net/specs/openid-connect-core-1_0.html#UserInfoError
  (and RFC 6750 — Bearer Token Usage, Section 3 "WWW-Authenticate Response Header Field")

According to the OIDC Core 1.0 spec, the UserInfo endpoint MUST return errors
using the WWW-Authenticate response header along with the appropriate HTTP
status code, with error codes drawn from RFC 6750:
  - invalid_request   -> HTTP 400
  - invalid_token     -> HTTP 401
  - insufficient_scope-> HTTP 403
And, for internal failures, a standard HTTP 500 with a meaningful body.


EXPECTED BEHAVIOUR (what the userinfo endpoint MUST return)
===========================================================
1. Any failure related to the access token (missing, malformed, expired,
   signature mismatch, unknown/no cached transaction, DPoP-binding failure,
   etc.):
        HTTP 401 Unauthorized
        WWW-Authenticate: Bearer error="invalid_token", error_description="..."

2. Client has requested an encrypted (JWE) userinfo response but the
   client's encryption public key / certificate is not configured or is
   invalid (i.e., the request itself cannot be honoured as asked):
        HTTP 400 Bad Request
        WWW-Authenticate: Bearer error="invalid_request", error_description="..."
   (Per the user's requirement this should be surfaced as `invalid_request`
   — NOT the internal `invalid_public_key` code, and NOT a generic 401.)

3. Any other internal/unhandled exception (crypto failure, plugin failure,
   NPE, cache failure, etc.):
        HTTP 500 Internal Server Error
        Body / header should expose error="internal_error" with a safe
        error_description (no stack traces, no internal codes leaked).


OBSERVED BEHAVIOUR (what the code currently does)
=================================================
Files involved:
  - esignet-service/src/main/java/io/mosip/esignet/advice/ExceptionHandlerAdvice.java
  - esignet-service/src/main/java/io/mosip/esignet/controllers/OpenIdConnectController.java
  - oidc-service-impl/src/main/java/io/mosip/esignet/services/OpenIdConnectServiceImpl.java
  - oidc-service-impl/src/main/java/io/mosip/esignet/services/UserInfoResponseHelper.java
  - esignet-core/src/main/java/io/mosip/esignet/core/constants/ErrorConstants.java

1. The controller produces `application/jwt` and returns a raw String. There
   is no controller-level mapping of business exceptions to spec-compliant
   WWW-Authenticate headers.

2. ExceptionHandlerAdvice.handleExceptionWithHeader(...) is the single
   handler for the `/oidc/userinfo` path. It:
     - Defaults HTTP status to 401 Unauthorized for EVERY exception type
       not explicitly switched on.
     - Defaults errorCode to UNKNOWN_ERROR.
     - Only maps NotAuthenticatedException -> invalid_token,
       MissingRequestHeaderException -> missing_header,
       HttpRequestMethodNotSupportedException -> 405,
       HttpMediaTypeNotAcceptableException -> 406.
     - Does NOT have a branch for `EsignetException` at all.
     - As a result, when UserInfoResponseHelper.encryptJWSToJWE() throws
       `new EsignetException(INVALID_PUBLIC_KEY)`, the advice falls through
       to the default case, returns HTTP 401 with header
       `WWW-Authenticate: error="unknown_error"`, and the meaningful
       `invalid_public_key` errorCode is LOST before reaching the client.
     - The response body is empty (`ResponseEntity<Void>`), so Postman sees
       only a generic 401 with an unhelpful header.

3. Confirmed by the log line provided:
       EsignetException: invalid_public_key
       at UserInfoResponseHelper.encryptJWSToJWE(UserInfoResponseHelper.java:132)
   The exception is correctly raised by the service but is swallowed/
   re-labelled by the advice.

4. Other gaps vs. the spec:
     - Errors are emitted as `error="..."` instead of the RFC 6750 form
       `Bearer error="...", error_description="..."`.
     - No `error_description` is included.
     - Internal exceptions are not differentiated — they all become 401
       instead of 500 with `internal_error`.
     - There is no body payload, so non-browser clients (Postman, curl)
       cannot see the reason without inspecting headers.
